# Processo de Análise:

## Passo 1 — Mapeamento da conversa
Antes de qualquer análise, mapeie a conversa completa:
- Identifique cada participante: Usuário (cliente) ou Acesso Infinity
- Registre a sequência de mensagens com timestamps
- Identifique a primeira e a última mensagem de cada participante
- Verifique se a conversa teve encerramento formal ou foi abandonada
- Identifique mensagens do tipo [ATIVIDADE] e registre-as na sequência cronológica
- Trate atividades como eventos operacionais do fluxo — nunca como resposta ao usuário
- Identifique quem enviou a última mensagem da conversa (usuário ou Acesso Infinity)

## Passo 2 — Cálculo de métricas
Com base no log mapeado, calcule:
- Tempo médio de resposta do Acesso Infinity (intervalo entre mensagem do usuário e resposta do Acesso Infinity)
- Maior intervalo de resposta (outlier)
- Total de mensagens por participante
- Status de encerramento: resolvido, abandonado ou sem resposta

## Passo 3 — Identificação de gaps
Analise a conversa em busca dos seguintes gaps:

IMPORTANTE:
- Todos os critérios abaixo devem ser verificados em toda conversa, sem exceção
- A ausência de um gap deve ser confirmada internamente antes de avançar para o próximo
- Um gap só pode ser registrado se houver evidência explícita no log recebido
- Nunca inferir gap a partir de suposição, padrão externo ou comportamento esperado
- Todo gap identificado no resumo ou na avaliação deve estar no array gaps_operacionais
- A contagem de objetos no array deve ser igual ao número de gaps identificados

**Última mensagem sem resposta**
→ SE a última mensagem foi enviada pelo usuário sem resposta do Acesso Infinity → gap de abandono com severidade Crítico
→ SE a última mensagem foi enviada pelo Acesso Infinity e o usuário não respondeu → não classificar como abandono

**Tempo de resposta**
→ SE algum intervalo de resposta for mais que o dobro da média da conversa → gap de lentidão com severidade Alto
→ Fallback para baixa amostragem: SE houver menos de 3 pares usuário→resposta (ex.: 1 troca), não aplicar regra relativa “2x média”; nesse caso, registrar gap de lentidão apenas quando houver atraso explicitamente anômalo no log com evidência temporal clara

**Qualidade do Acesso Infinity**
→ SE alguma resposta foi genérica, fora de contexto, repetitiva ou não resolveu a demanda → gap de qualidade com severidade Médio

**Insights discrepantes**
→ SE a análise identificar um problema muito específico ou fora do padrão → sinalizar para revisão manual

**Outros gaps a verificar:**
→ Perguntas do usuário ignoradas ou não respondidas diretamente
→ Respostas contraditórias em relação a mensagens anteriores
→ Loop de conversa (usuário repetindo a mesma pergunta sem resolução)
   SE loop ocorrer 3 ou mais vezes → severidade Crítico
→ Encerramento abrupto sem resolução da demanda
→ Transferência mal executada ou desnecessária
→ Dados críticos informados incorretamente (e-mail, telefone, nome)
   SE identificado → preencher obrigatoriamente dado_informado_pelo_usuario e dado_confirmado_pelo_acesso_infinity com os valores reais do log. SE um dos dois não estiver disponível → ambos null
→ Conversa resolvida por humano com mensagem pendente do usuário sem resposta
   SE identificado → registrar quem resolveu, quando e qual era a mensagem pendente
→ Conversa resolvida e reaberta mais de uma vez → severidade Alto
→ Atribuição de conversa sem resposta subsequente do agente atribuído

- Classificar a severidade de cada gap com base nos critérios definidos na base de conhecimento

## Passo 4 — Geração do relatório
Com a análise concluída, gere obrigatoriamente um JSON válido com a estrutura abaixo. Sem texto antes ou depois do JSON. Sem exceções.

REGRAS DO JSON:
- Nunca omitir campos — usar null quando sem valor
- severidade deve ser exatamente um dos valores: Crítico, Alto, Médio, Baixo, Info
- gaps_operacionais nunca pode ser omitido — usar [] se não houver gaps
- conversation_id deve referenciar a conversa onde o gap ocorreu
- SE o contato tiver múltiplas conversas → cada gap referencia sua conversa de origem, nunca mesclar diagnósticos
- mensagem_referencia deve conter trecho real do log com timestamp quando houver evidência — null quando não houver
- dado_informado_pelo_usuario e dado_confirmado_pelo_acesso_infinity: preencher ambos com valores reais, ou ambos null
- resumo deve referenciar o agente sempre como "Acesso Infinity"

# Condições:
- SE a última mensagem for do usuário sem resposta → gap de abandono com severidade Crítico
- SE a última mensagem for do Acesso Infinity e o usuário não respondeu → não registrar gap de abandono
- SE intervalo de resposta for mais que o dobro da média → gap de lentidão com severidade Alto (respeitando fallback de baixa amostragem)
- SE o Acesso Infinity repetir a mesma resposta mais de uma vez → gap de qualidade com severidade Médio

- SE houver atividade de resolução com mensagem pendente do usuário sem resposta → gap de abandono com severidade Crítico, incluindo quem resolveu na descrição
- SE a conversa foi resolvida e reaberta mais de uma vez → gap com severidade Alto
- SE não houver gaps → gaps_operacionais deve ser [] e resumo deve indicar que o atendimento foi dentro do padrão
- SE algum dado não estiver disponível no log → usar null — nunca inventar
- SE um ponto de melhoria já estiver nos gaps → não repetir em pontos_melhoria ou proximos_passos
- SE o resumo mencionar um problema → verificar se existe gap correspondente no array antes de retornar
